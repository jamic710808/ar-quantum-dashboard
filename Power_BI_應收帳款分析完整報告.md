# 一、執行摘要

本報告彙整資料夾內六份 Power BI 應收帳款（AR）分析技術文件的核心內容，涵蓋思維框架、數據建模、DAX 公式全集、視覺設計、風險管控模型，以及系統性的優化方向與設計新思維建議。

應收帳款分析的本質是「信用投資管理」：每一筆賒銷都是一項投資，目標是最大化回報率（按時回收）、最小化機會成本（信用資金佔壓）。傳統單維度指標（賒銷比率、帳齡天數、違約率）均存在致命盲點，無法全面反映風險的頻率、金額、時間三個維度。本系統以「信用資金成本率」為核心旗艦指標，實現了從靜態快照到動態風險演進的典範轉移。

# 二、分析思維框架重塑

## 2.1 傳統分析局限與三大盲點

傳統應收帳款分析依賴三種片面指標，每種均存在系統性盲點：

* **盲點一：賒銷比率（Write-off Ratio）：**誤區：賒銷比率 50% ≠ 高風險。案例：昆明某貿易公司累計營收 44 萬、期末應收 22 萬，賒銷比率高達 50%，但 4 次交易均按時結算，帳齡天數為零，實際無風險。賒銷比率只反映賒銷規模，不反映違約行為。
* **盲點二：帳齡天數（Aging Days）：**誤區：帳齡天數的「動態清零」特性。客戶昨天逾期 365 天，今天還款，帳齡立即歸零，徹底掩蓋了歷史違約行為。靜態帳齡無法呈現信用歷史的全貌。
* **盲點三：違約率（Default Rate）：**誤區：同樣 25% 違約率，風險差異天壤之別。客戶 A：違約金額 2 萬元、逾期 30 天；客戶 B：違約金額 20 萬元、逾期 90 天。違約率只考慮頻率，忽略金額（Amount）與時間（Duration）。

## 2.2 三維風險模型

真正的風險是三維的：頻率（Frequency）× 金額（Amount）× 時間（Duration）。任何忽略其中一個維度的指標都是不完整的。信用資金成本率同時量化這三個維度，是唯一能全面衡量客戶信用風險的綜合指標。

## 2.3 授信即投資：思維重塑

應收帳款的核心邏輯是「信用投資」：每一筆授信都是公司對客戶的投資，收益是未來的銷售收入，成本是資金佔壓的機會成本。分析目標從「控制壞帳」升級為「最佳化授信資源配置」——將有限的信用額度，集中投資在回報率最高（風險最低）的客戶身上。

# 三、完整分析架構解析

## 3.1 五維量化指標體系

所有 AR 指標歸納為五個維度，確保分析全面且結構化：

| --- | --- | --- | --- |
| **維度** | **中文名** | **核心指標** | **業務意義** |
| Scale | 規模 | 期末應收餘額、累計營收 | 衡量應收帳款的絕對量級，判斷資金佔壓程度 |
| Efficiency | 效率 | 應收帳款周轉率、DSO | 衡量資金回收速度，越快越好 |
| Ratio | 比率 | 應收帳款比率、逾期率 | 應收與業務規模的相對關係，評估賒銷政策 |
| Aging | 帳齡 | 加權平均帳齡、帳齡結構佔比 | 深度分析欠款時間分佈，識別老齡化應收 |
| Cost | 成本 | 信用資金成本、信用資金成本率 | 量化應收帳款的機會成本，是風險決策的核心 |

## 3.3 星型數據模型（Star Schema）

數據模型採用標準星型架構（Star Schema），70% 的系統價值在於正確的建模設計：

* **事實表 1：應收表：**記錄所有賒銷發票，包含發票編號、客戶代碼、發票金額、開票日期、到期日。每筆賒銷交易一行。
* **事實表 2：實收表：**記錄所有實際收款記錄，包含收款日期、收款金額、對應發票/客戶。每筆收款一行。
* **維度表：客戶表：**客戶主數據：客戶代碼、客戶名稱、所屬城市、所屬業務員、信用額度、信用期限。
* **維度表：日期表：**標準日期維度表，包含年/季/月/日、是否工作日等屬性。是時間智慧函數的基礎。
* **維度表：城市表：**城市/地區層級，用於地理分析。
* **維度表：業務員表：**業務員主數據，用於業績歸因與責任劃分分析。
* **孤島表：帳齡分類輔助表：**無物理關聯的特殊維度表，欄位：序號、分類名稱、最小天數、最大天數、預計壞帳損失率。用於動態帳齡分桶（Disconnected Slicer Pattern）。

## 3.4 三大核心計算方法

* **方法一：迭代計算（SUMX Pattern）：**使用 SUMX 在每一行的行上下文（Row Context）中逐筆計算，再彙總。適用場景：加權平均帳齡計算（每張發票的餘額 × 帳齡天數後求和）、預計壞帳損失（每個帳齡分類的餘額 × 壞帳率後求和）。
* **方法二：反向篩選（Disconnected Table / Reverse Filter Pattern）：**孤島表無物理連結，藉由 CALCULATE + FILTER 將孤島表的篩選值反向施加給事實表。適用場景：動態帳齡分桶、動態區間篩選。優勢：不修改數據模型即可支援使用者動態切換分組方式。
* **方法三：上下文操控（CALCULATE Context Manipulation）：**使用 CALCULATE 重新定義篩選上下文（Filter Context），搭配 ALL、ALLEXCEPT、FILTER 進行精確的上下文控制。適用場景：時間截止計算、跨維度比率計算、解除特定篩選的彙總計算。

# 四、DAX 公式完整彙整

以下收錄本系統全部 15 個核心 DAX 量值（Measure）。每個公式包含完整語法、計算邏輯說明、行/篩選上下文說明。

**【重要提示】所有財務金額欄位（發票金額、收款金額等）必須在 Power Query 中設定為「定點小數（Fixed Decimal Number）」類型，避免浮點誤差導致帳齡計算錯誤（如本應為 0 的餘額顯示為 0.000000000000002）。**

**公式 01　期末應收餘額**　[規模指標]

期末應收餘額 =

CALCULATE(

SUM('應收表'[發票金額]) - SUM('實收表'[收款金額]),

FILTER(

ALL('日期表'),

'日期表'[Date] <= MAX('日期表'[Date])

)

)

**說明：**核心基礎指標。以累計應收金額減去累計實收金額，搭配 FILTER(ALL(日期表)) 實現時間截止邏輯，確保計算的是「截至所選日期為止」的未收餘額，而非單期差額。CALCULATE 用於覆蓋篩選上下文（Filter Context）。

**⚙ 上下文：**ALL('日期表') 解除日期篩選後重新施加 <= 條件，避免交叉篩選導致的截止誤差。

**公式 02　單筆訂單帳齡（發票層級）**　[帳齡指標]

單筆訂單帳齡 =

IF(

[應收餘額，期末] > 0,

DATEDIFF('發票資訊'[到期日], TODAY(), DAY),

0

)

**說明：**在發票資訊表的行上下文（Row Context）中計算每張發票的逾期天數。只有餘額 > 0 時才計算帳齡，已結清的發票返回 0，避免歷史逾期污染當前指標。

**⚙ 上下文：**此為計算資料行（Calculated Column），在行上下文中執行。

**公式 03　加權平均帳齡**　[帳齡指標]

加權平均帳齡 =

VAR TotalReceivables = [應收餘額，期末]

VAR WeightedSum =

SUMX(

VALUES('發票資訊'[發票編號]),

[應收餘額，期末] \* [單筆訂單帳齡]

)

RETURN

DIVIDE(WeightedSum, TotalReceivables)

**說明：**迭代計算（SUMX）精髓：以每張發票的期末應收餘額為權重，加權計算平均帳齡天數。公式 = Σ(單筆餘額 × 帳齡天數) / 總餘額。DIVIDE 避免分母為零，相較於 MAX 帳齡更能反映資金佔壓的真實程度。

**⚙ 上下文：**SUMX 在每張發票的行上下文中分別計算後求和，是迭代計算（Iteration）模式的典型應用。

**公式 04　帳齡管控分類餘額**　[帳齡指標]

帳齡管控分類餘額 =

VAR MinDays = MIN('帳齡分類輔助表'[最小值])

VAR MaxDays = MAX('帳齡分類輔助表'[最大值])

RETURN

CALCULATE(

[應收餘額，期末],

FILTER(

'發票資訊',

[加權平均帳齡] >= MinDays && [加權平均帳齡] < MaxDays

)

)

**說明：**孤島表（Disconnected Table）模式核心。帳齡分類輔助表與事實表無物理關聯，透過反向篩選（Reverse Filter）：從輔助表讀取當前篩選區間的最小/最大天數，再對發票資訊表施加條件。此設計允許使用者在視覺物件中動態切換帳齡分組，無需修改數據模型。

**⚙ 上下文：**孤島表篩選上下文（Disconnected Slicer Pattern）：輔助表的篩選值藉由 MIN/MAX 函數傳遞給 FILTER，實現動態分桶。

**公式 05　帳齡結構佔比**　[帳齡指標]

帳齡結構佔比 =

DIVIDE(

[帳齡管控分類餘額],

CALCULATE([期末應收餘額], ALL('帳齡分類輔助表'))

)

**說明：**計算各帳齡段佔總應收餘額的百分比。分母使用 ALL('帳齡分類輔助表') 移除帳齡段篩選以獲得總餘額，確保各段佔比加總為 100%。適用於堆疊面積圖呈現帳齡結構趨勢。

**⚙ 上下文：**ALL() 解除特定維度篩選的標準用法，與 ALLEXCEPT 形成互補。

**公式 06　預計壞帳損失額**　[風險指標]

預計壞帳損失額 =

SUMX(

'帳齡分類輔助表',

[帳齡管控分類餘額] \* '帳齡分類輔助表'[預計壞帳損失率]

)

**說明：**跨帳齡段的迭代計算：對每個帳齡分類，取該分類的期末餘額乘以對應壞帳準備率，然後加總。壞帳準備率設定：未超信用期 0%、1-30天 1%、31-90天 3%、91-365天 5%、365天以上 20%。

**⚙ 上下文：**SUMX 在帳齡分類輔助表的每一行產生上下文轉換，使 [帳齡管控分類餘額] 的計算自動對應到該行的分類範圍。

**公式 07　信用資金成本**　[成本指標]

信用資金成本 =

[期末應收餘額] \* [加權平均帳齡] \* (0.06 / 365)

**說明：**將應收帳款量化為機會成本：應收餘額 × 加權平均帳齡 × 日利率。日利率 = 年化利率（如 6%）/ 365，反映客戶拖欠帳款期間公司損失的潛在投資收益。此指標同時涵蓋帳款金額、逾期時間兩個維度。

**⚙ 上下文：**年利率可依公司資金成本或市場基準利率調整，建議設為參數化量值以方便維護。

**公式 08　信用資金成本率（核心風險指標）**　[成本指標]

信用資金成本率 =

DIVIDE(

[信用資金成本],

CALCULATE(

SUM('銷售表'[銷售金額]),

FILTER(

ALL('日期表'),

'日期表'[Date] <= MAX('日期表'[Date])

)

)

)

**說明：**終極風險量化指標，整合頻率（違約次數）、金額（應收餘額）、時間（帳齡天數）三個維度。分子為信用資金成本，分母為累計營收。關鍵特性：動態自我修正——客戶只要持續良好履約，分母（累計營收）不斷增大，即使歷史污點不變，比率也會逐漸下降，實現動態風險評級。

**⚙ 上下文：**此指標超越靜態帳齡天數和孤立違約率，是「頻率 × 金額 × 時間」三維風險的完整映射。

**公式 09　風險等級分類**　[風險指標]

風險等級 =

SWITCH(TRUE(),

[信用資金成本率] = 0, "無風險",

[信用資金成本率] < 0.001, "無風險",

[信用資金成本率] < 0.005, "低風險",

[信用資金成本率] < 0.010, "中風險",

[信用資金成本率] < 0.020, "高風險",

"超高風險"

)

**說明：**基於信用資金成本率的五級動態風險分類。SWITCH(TRUE()) 模式從第一個成立的條件返回結果，是 DAX 中實現多條件分支的最佳實踐。閾值：無風險 <0.1%、低風險 0.1%-0.5%、中風險 0.5%-1.0%、高風險 1.0%-2.0%、超高風險 ≥2.0%。

**⚙ 上下文：**應作為計算資料行建立在客戶維度表或使用 SUMMARIZE 建立的摘要表中，以支援視覺化篩選。

**公式 10　應收帳款周轉率（優化版）**　[效率指標]

應收帳款周轉率 =

DIVIDE(

CALCULATE(

SUM('實收表'[收款金額]),

DATESINPERIOD('日期表'[Date], LASTDATE('日期表'[Date]), -12, MONTH)

),

DIVIDE(

[期末應收餘額] + CALCULATE([期末應收餘額], DATEADD('日期表'[Date], -12, MONTH)),

2

)

)

**說明：**優化重點：分子使用年度現金實收金額（而非銷售收入），更準確反映資金回收效率；分母使用期初期末平均應收餘額。傳統錯誤做法是用賒銷收入作分子，無法衡量實際收款效率。

**⚙ 上下文：**DATESINPERIOD 計算滾動12個月收款，DATEADD 取12個月前的期末餘額作為期初值。

**公式 11　應收帳款周轉天數（DSO）**　[效率指標]

應收帳款周轉天數 =

DIVIDE(

[期末應收餘額] \* 365,

CALCULATE(

SUM('實收表'[收款金額]),

DATESINPERIOD('日期表'[Date], LASTDATE('日期表'[Date]), -12, MONTH)

)

)

**說明：**Days Sales Outstanding，衡量應收帳款平均回收天數。DSO越低代表資金回收效率越高。計算邏輯：期末應收餘額 / 日均收款金額。與加權平均帳齡形成互補：DSO 衡量整體效率，加權平均帳齡衡量單客戶/發票維度的帳款老化程度。

**⚙ 上下文：**建議同時展示 DSO 趨勢圖與行業基準比較，使管理層能快速識別效率惡化信號。

**公式 12　應收帳款比率**　[比率指標]

應收帳款比率 =

DIVIDE(

[期末應收餘額],

CALCULATE(

SUM('銷售表'[銷售金額]),

FILTER(ALL('日期表'), '日期表'[Date] <= MAX('日期表'[Date]))

)

)

**說明：**反映應收帳款佔累計銷售收入的比例，衡量賒銷政策的鬆緊程度。比率過高表示大量收入尚未回收，資金壓力大；比率過低則可能喪失市場競爭力（過於嚴格的收款政策）。

**⚙ 上下文：**分母使用累計營收（含 FILTER/ALL 邏輯），與 [信用資金成本率] 的分母邏輯保持一致。

**公式 13　逾期率**　[風險指標]

逾期率 =

DIVIDE(

CALCULATE(

[期末應收餘額],

FILTER('發票資訊', [單筆訂單帳齡] > 0)

),

[期末應收餘額]

)

**說明：**計算已超過信用期的應收餘額佔總應收餘額的比例。注意：逾期率只考慮「頻率」維度，忽略金額大小和逾期時間長短，是片面指標。同樣 25% 逾期率，逾期金額 2 萬和 20 萬，逾期 30 天和 90 天，風險差異極大，因此須結合信用資金成本率使用。

**⚙ 上下文：**逾期率僅為輔助參考指標，不應單獨作為風險決策依據。

**公式 14　累計營收（時間截止版）**　[規模指標]

累計營收 =

CALCULATE(

SUM('銷售表'[銷售金額]),

FILTER(

ALL('日期表'),

'日期表'[Date] <= MAX('日期表'[Date])

)

)

**說明：**計算截至所選日期的累計銷售收入，作為信用資金成本率的分母。時間截止邏輯確保不同時期比較時，分母會隨業務累積而增大，體現動態風險遞減效果。

**⚙ 上下文：**此公式與 [期末應收餘額] 的時間邏輯完全平行，保證兩者在同一時間截面上可比較。

**公式 15　平均應收帳款餘額**　[規模指標]

平均應收帳款餘額 =

DIVIDE(

[期末應收餘額] +

CALCULATE(

[期末應收餘額],

DATEADD('日期表'[Date], -1, YEAR)

),

2

)

**說明：**期初期末平均，作為周轉率計算的分母，減少期末單點數據的偶然性波動。DATEADD 偏移 -1 年取得期初值，與標準財務計算公式一致。

**⚙ 上下文：**若需要更精確的月度平均，可使用 AVERAGEX + DATESINPERIOD 計算月末餘額的平均值。

# 五、五級風險分類矩陣與應對策略

基於信用資金成本率，將客戶分為五個風險等級，每級對應明確的授信管理策略，實現「量化驅動的精準信用管理」：

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| **風險等級** | **資金成本率閾值** | **客戶特徵** | **核心策略** | **行動要點** |
| ✅ 無風險 | < 0.1% | 信用記錄優異，履約表現良好 | 提升授信 | 可適度提高賒銷比例、延長帳期，作為優質客戶獎勵 |
| ✓ 低風險 | 0.1%-0.5% | 信用記錄良好，偶有輕微逾期 | 維持現狀 | 保持現有授信政策，持續觀察，每季複查一次 |
| ⚠ 中風險 | 0.5%-1.0% | 出現違約苗頭，違約次數或金額上升 | 加強催收 | 及時發送催款通知，暫停新的賒銷業務直到帳款結清 |
| ⚠ 高風險 | 1.0%-2.0% | 違約記錄嚴重，帳齡長，金額較大 | 立即清收 | 立即採取行動催收，清帳後重新評估並考慮取消授信 |
| ❌ 超高風險 | ≥ 2.0% | 履約記錄極差，對公司構成顯著威脅 | 現款結算 | 清帳後取消所有授信，未來訂單一律要求現款交易 |

## 5.1 信用資金成本率的動態特性

資金成本率 = 信用資金成本（固定分子）÷ 累計營收（變動分母）

這個設計具有獨特的動態自我修正機制：客戶只要持續保持良好的履約記錄，分母（累計營收）會不斷變大。即使分子（歷史污點）不變，整體「資金成本率」也會逐漸下降，風險等級隨之動態下調。

這不僅是一個風險評估模型，更是一個客戶關係管理工具——它為每個客戶提供了一條通過良好行為改善信用評級的清晰路徑。

# 七、現有設計優化方向

## 7.1 數據類型優化（高優先級）

|  |
| --- |
| **🚨 最常見的致命錯誤**  財務金額欄位若使用「十進位數（Decimal Number）」類型，Power BI 可能儲存浮點誤差（如 0.000000000000002），顯示為 0.00 但計算時不等於零，導致帳齡天數計算錯誤。必須改用「定點小數（Fixed Decimal Number）」類型並設定 2 位小數。 |

* 所有發票金額、收款金額、應收餘額等財務欄位 → 改用 Fixed Decimal Number（定點小數）
* 在 Power Query 中統一設定，不要在 DAX 中用 ROUND 修補（治標不治本）
* 設定完成後全量重新整理，確認所有近零值（< 0.01）均正確歸零

## 7.2 帳齡計算方法優化

* **從 MAX 帳齡改為加權平均帳齡：**傳統 MAX(帳齡天數) 只取最大值，一筆小額逾期發票會拉高整個客戶的帳齡指標，嚴重誤導風險判斷。改用加權平均帳齡（SUMX 模式），以金額為權重，真實反映資金佔壓的時間代價。
* **從靜態帳齡分桶改為動態分桶：**固定帳齡分類（如寫死「0-30天」區間）不靈活，難以支援管理層的臨時分析需求。改用孤島表（Disconnected Table）模式，支援使用者在切片器中動態調整分桶區間。
* **帳齡基準日設計：**帳齡計算應以「到期日」為基準（而非開票日），才能真實反映逾期程度。到期日 = 開票日 + 信用期限，需確保客戶維度表中有正確的信用期限設定。

## 7.3 周轉率計算公式優化

【傳統錯誤】應收帳款周轉率 = 銷售收入 / 平均應收帳款餘額

**【優化正確】應收帳款周轉率 = 年度現金回款金額 / 平均應收帳款餘額**

理由：分子使用「銷售收入」無法反映實際收款效率，即使有大量逾期帳款，周轉率依然好看。改用「現金回款金額（實收表）」才能真正衡量資金回收速度，是準確的效率指標。

## 7.4 風險指標體系優化

* **從四級改為五級風險分類：**原四級（無/低/中/高）增加「超高風險」級別，更精確識別需要立即採取現款結算的危險客戶。
* **引入信用資金成本率替代違約率：**違約率是一維指標（只看頻率），信用資金成本率是三維指標（頻率×金額×時間）。後者能揭示違約率相同但風險完全不同的客戶差異。
* **動態風險評級機制：**資金成本率的設計讓風險評級自動隨客戶行為動態調整，避免「一次違約、永遠高風險」的靜態懲罰邏輯，為客戶提供信用修復的激勵機制。

## 7.5 數據模型結構優化

* 確保應收表和實收表可以通過發票編號或客戶代碼正確關聯，避免多對多關係
* 日期表必須連續、完整（無缺漏日期），且標記為 Power BI 日期表，以啟用時間智慧函數
* 帳齡分類輔助表確保為孤島表（無任何物理關聯），才能正確實現 Disconnected Slicer 模式
* 建議在模型中建立獨立的「參數表」存放可調整參數（如年利率、壞帳準備率），避免硬編碼在 DAX 中

# 八、設計架構新思維與創新建議

## 8.1 從靜態快照到滾動式動態監控

現有設計以月末/期末快照為主。建議引入「滾動窗口（Rolling Window）」設計概念：

* 滾動 3 個月違約趨勢：計算每個客戶近 3 個月的平均資金成本率，識別風險上升趨勢
* 滾動 DSO 趨勢：以 12 個月滾動窗口計算 DSO，排除季節性波動影響，呈現真實的效率趨勢
* 預警信號層：設定閾值觸發規則，當客戶資金成本率從上月到本月上升超過 0.3% 時，自動標記為「風險上升」狀態

## 8.2 客戶生命週期視角

建議增加「客戶信用生命週期」分析維度：

* 新客戶期（首 3 筆交易）：重點觀察首次交易是否按時結算，設置較嚴格的信用額度
* 成長期（3-10 筆交易）：評估履約穩定性，根據歷史表現動態調整授信額度
* 成熟期（10 筆以上）：基於長期資金成本率評分，給予差異化授信政策
* 衰退期（連續 2 期風險等級上升）：自動觸發減少授信或要求擔保機制

## 8.3 主動式催收決策引擎

從「被動報表」升級為「主動決策引擎」：

* **優先催收隊列（Priority Collection Queue）：**根據 [預計壞帳損失額 × 逾期天數] 計算每個客戶的「催收緊迫指數」，自動排序，讓業務團隊每天打開報表就知道今天優先催收哪些客戶
* **模擬分析（What-If）：**使用 Power BI 參數功能，讓管理層模擬「如果某客戶在 X 天後還款，資金成本率會下降多少」的情境，量化催收行動的財務收益
* **預測帳齡（Forecast Aging）：**基於歷史還款行為模式，預測未來 30/60/90 天的帳齡分佈，提前識別可能超齡的高風險發票

## 8.4 多維度客戶評分體系

建議建立綜合客戶信用評分（Credit Scoring）模型，整合多個維度：

|  |  |  |  |
| --- | --- | --- | --- |
| **評分維度** | **指標** | **權重建議** | **DAX 基礎** |
| 資金佔壓成本 | 信用資金成本率 | 40% | 公式08 |
| 帳款老化程度 | 加權平均帳齡 | 25% | 公式03 |
| 潛在損失風險 | 預計壞帳損失額/應收餘額 | 20% | 公式06 |
| 資金回收效率 | DSO 相對行業基準 | 15% | 公式11 |

## 8.5 授信資源配置最佳化

以「授信資源如同投資組合」的理念，建立授信配置矩陣：

* 高業績 + 低風險客戶：主動提高授信額度和帳期，激勵業績進一步增長
* 高業績 + 高風險客戶：維持業務但要求分批付款或預付款，降低單次風險敞口
* 低業績 + 低風險客戶：維持現狀，探索業績增長潛力
* 低業績 + 高風險客戶：優先清收，考慮停止賒銷業務，是授信資源最差的配置

## 8.6 Power BI 技術架構升級建議

* **DAX 參數化設計：**將年利率（6%）、壞帳準備率等可調參數建立為 What-If Parameter 或獨立參數表，支援場景模擬而無需修改 DAX 公式
* **增量刷新（Incremental Refresh）：**對大型應收/實收事實表啟用增量刷新，大幅降低每日數據更新時間，只刷新新增/修改的記錄
* **Row-Level Security（RLS）：**基於業務員或城市維度設定行級安全性，確保業務人員只看到自己負責的客戶數據
* **Composite Model + DirectQuery：**對需要即時數據的場景，考慮對實收表使用 DirectQuery 模式，確保收款信息實時反映
* **計算群組（Calculation Groups）：**統一管理時間智慧邏輯（YTD、MTD、滾動12個月），避免在每個量值中重複編寫時間截止邏輯

# 九、實務技術重點彙整

## 9.1 常見錯誤與修正

|  |  |  |
| --- | --- | --- |
| **錯誤類型** | **錯誤做法** | **正確做法** |
| 數據類型 | 財務欄位用 Decimal Number | 改用 Fixed Decimal Number（定點小數） |
| 帳齡計算 | 用 MAX(帳齡) 作客戶帳齡指標 | 改用 SUMX 計算加權平均帳齡 |
| 周轉率分子 | 分子用銷售收入 | 分子改用現金回款金額（實收表） |
| 風險評級 | 用孤立違約率決定風險 | 改用信用資金成本率（三維指標） |
| 帳齡分桶 | 在 DAX 硬編碼固定分桶 | 改用孤島表 + Disconnected Slicer 動態分桶 |
| 參數管理 | DAX 中硬編碼利率數值 | 建立 What-If Parameter 或參數表 |
| 日期表 | 未標記為官方日期表 | 在 Power BI 中標記為官方日期表以啟用時間智慧函數 |

## 9.2 DAX 效能最佳實踐

* 優先使用 DIVIDE() 代替 "/" 運算符，自動處理除以零的情況，避免產生 Error 值
* 在量值中使用 VAR … RETURN 結構，避免重複計算相同的中間結果
* 避免在量值中使用 RELATED()（會觸發行上下文），改在計算資料行或使用 RELATED() 在正確的行上下文中
* 使用 VALUES() 而非 DISTINCT() 作為 SUMX 的迭代表，VALUES() 會自動包含空白行處理
* 複雜的篩選邏輯優先使用 CALCULATETABLE() 建立虛擬表，再在虛擬表上聚合，比多層巢狀 FILTER 效能更好
* 對大型事實表，避免在量值中使用 COUNTROWS(FILTER()) 模式，改用更高效的 CALCULATE(COUNT(), filter conditions)

## 9.3 建模最佳實踐

* 日期表必須包含報表所需的最早日期到最晚日期的連續完整日期，並在 Power BI 中標記為日期表
* 事實表與維度表之間只建立多對一（Many-to-One）的單向關聯，避免雙向篩選（除非有充分理由）
* 孤島表（帳齡分類輔助表）不設定任何關聯，讓 FILTER 在 DAX 中完成動態連結
* 所有描述性屬性（如客戶名稱、城市名稱）存放在維度表，不要存放在事實表中
* 事實表只保留外鍵（ID）和量值（金額、數量），不存放維度描述性欄位
* 在 Power Query 中完成所有數據清洗和轉換，不要在 DAX 中做數據清洗工作
